Michigan Recreation

The Michigan Department of Natural Resources is responsible for managing and maintaining all of the state parks, forests, trails, beaches, and other state-owned property. As a service to residents, the department offers a "Recreation Search" tool which allows users to search for hiking trails, campgrounds, state parks, hunting areas, etc.

As an avid hiker, I use this tool quite a bit to find trails and campgrounds for backpacking. I am consistently frustrated by how difficult it can be to find information with it — my experiences teetering on the edge of unhelpful and completely useless. Often times, I find it easier to use other online sources to find information I need.

Thus, I decided to try my hand at improving the DNR (Department of Natural Resources) recreation search tool. How can this be easier to use? How can it provide better information? To answer those questions, I began a personal passion project: to fix the worst part of camping/hiking in Michigan.

1.0 Research

1.1 Identifying Scope

The recreation search tool is a large website with thousands of pages. The scope of the project was narrowed to redesigning the primary pages and user flow for a hiker looking for a trail.


1.2 Problems

After conducting a heuristic evaluation and an expert review, there were several issues identified which were detrimental to the user experience of the application.

Inconsistent with Michigan design guidelines

Recreation tool homepage vs. Michigan.gov homepage

A common path for discovering the Recreation Search Tool is through the Michigan.gov website. The issue lies in the seemingly random change in design systems between the two websites. The Recreation Search Tool completely breaks any resemblance to the Michigan.gov website, while also breaking State of Michigan design guidelines for the web. The redesigned tool must adhere to the guidelines set forth for digital applications to improve user experience, learnability, and usability, as well as align with the Michigan brand.

Uninformative search results

Trail search results on current website

When the user is looking for a trail to hike on or a place to camp, there are typically parameters. If a user wants to do a two day backpacking trip over the weekend, this user probably doesn't want to drive 9 hours to get to the park. This user also is not going to be looking for trails 50+ miles long, nor trails 8 miles long. This important information is not displayed when the user is scanning the trail search results, which means they would have to click on each trail which looks interesting.

Confusing reservation system

When you click on "Make a Reservation" you are taken to a seperate website. The campground you selected did not auto-fill on the new website.

Most recreational activities require a reservation, however the reservation system in-place adds unnecessary burden on the user. When a user finds a campground they would like to go to and clicks "Make a Reservation", they will be taken to a completely seperate website. What's worse is that the field "state park" doesn't autofill when the user arrives. This means the user now has to think actively about what park and campground they wanted, as well as add additional actions the user must take to complete a task they already completed.

1.3 Goals

  1. Overhaul UI. Redesign application for usability and adhere to Michigan UI guidelines.
  2. Provide information when it's needed. On all pages, information should be helpful and easy to find.
  3. Reduce user actions. Tasks should be able to be performed with as few clicks as possible.
  4. Changes should be feasible. Although this is a passion project, all changes to the tool should not require drastic alterations to underlying infrastructure.

2.0 Design

2.1 Ideation

Before I began designing, I created a user flow diagram detailing all the actions a user might take while searching for campground and things to do while camping. I then identified key points in the flow that could be simplified. From there, low-fidelity paper sketches were created to guide the rest of the design process. The paper prototypes brought several revisions of content structure, and specifically encouraged consideration for the end-to-end user experience. In this, determinations were made about cross-page navigation in different user contexts and paths.

Some paper prototype screens

2.2 Wireframes

The next step was wireframing. We were going onto week two of the project, and this was where most of the design decisions were made. This also led to a variety of problem solving: how can we easily help users find where to go? How can we simplify searching? How can we organize content on each page? Most of all, how can all of this be done while meeting the goals of the project and being technically feasible?

Searching

Trail search results on wireframe

To make scanning easier, additional information was included on each search result. Now, if the data is available, the name, brief description, activity list, distance, region, trail type, and available campsites would be listed. This is compared to before, when only the brief descriptin and activity list was included.

Trail Pages

Trail details page

The new content page considers hierarchy in a more pronounced way. Important information is with the rest of the content, and is larger than the rest of the text for easy scannability. Other information is included on the page, like on-trail campgrounds and connecting trails. This information is not included on the current page, but would assist hikers in planning for trips. Additionally, instead of hiding information behind other tabs, the trail map and directions map would be at the bottom of the page. This way, information needed when browsing is at the top and the information needed when making detailed plans is available near the bottom. This is true across wireframes.

Reservation Page

The new reservation page

This page does not currently exist in the tool. On the original Recreation Search Tool, the user must exit and go to a different website to make reservations. After the redesign, all of that information is collected on a form in the Recreation Search Tool reservation page, sends that information to the reservation website, and then populates the page with an iFrame of the reservation website result. This is a band-aid solution to the cross-website mess, but is a cheap solution. Ideally, this would all happen natively in the Recreation Search Tool.

Trip Planner Tool

The new trip planner page

This page is also a new addition and is essentially a cleaner filter page. The user selects what they want to do (backpack, camp, fish, etc.), where in Michigan they want to travel to, other information which is specific to what they are doing (such as trail length), and trip length. The tool will spit out a filtered list of all trails, parks, campgrounds, hunting areas, marinas, etc. which meet the criteria. This is a conversational approach to filtering, and minimizes cognitive load for the user.


All wireframes

2.3 Prototypes

After a user test with a hiker, a few revisions were made to the wireframes. In the final week of the project, hi-fidelity prototype screens were crafted. During the process of converting the wireframes into hi-fidelity screens, decisions about visual elements such as colors, drop shadows, button designs, and icons were made. Limited interactions were also implemented.

Final Search Page

Hi-fidelity screen of the search page

A change implemented after user feedback was changing the information which is displayed to "completion time" from "on-trail campgrounds." This change provides more useful information to many hikers. Time commitment is a driving factor behind day trips and is far more important to have easily accessible than if there are campgrounds along the trail.

Final Trail/Park Page

A hi-fidelity state park screen; similar to trail pages, campground pages, etc.

The final screens improve upon the current state park pages in three ways:

  1. Easier to scan. Important information is near the top and is larger. Detailed information is found at the bottom. More whitespace is introduced compared to before.
  2. Visually pleasing. Previous pages felt slightly chaotic, had little prominance, and lacked an engaging design. The redesign delegates images to become hero-images, removes tabs, adds color, and puts content into a two-column grid layout.
  3. On-page reservations. Now users do not have to leave the page to make reservations, simplifying the process of making a reservation by eliminating the need to go to a different page.

On-page reservation system example

Final Reservation Page

The new reservation page

Nearly unchanged from the wireframes, the new reservation page centralizes the entire process of making a reservation onto one page rather then spread across several websites.

Hi-Fidelity Trip Planner Tool

The hi-fidelity trip planner screens

A conversational approach to filtering, and minimizes cognitive load for the user. The page features detailed county-level regions which ease the search process. Once the form is submitted, a filtered list is returned of all activities and parks that meet the user criteria.
